## 1. Embedding 模型

### 是什么
把文字转成高维数字向量的模型。语义相近的句子，在向量空间里距离也相近。

### 三类模型对比
| 模型 | 维度 | 特点 | 适用场景 |
|------|------|------|---------|
| all-MiniLM-L6-v2 | 384维 | 英文优化，轻量 | 英文场景 |
| BGE-M3 | 1024维 | 中英文双语，开源 | 中文场景首选 |
| 智谱 embedding-3 | 2048维 | 中文优化，API调用 | 中文场景，无需本地部署 |

### 你的实验数据
以「如何提高机器学习模型的准确率」为基准：

| 对比句 | MiniLM | 智谱embedding-3 |
|--------|--------|----------------|
| 怎样让机器学习效果更好（语义相近） | 0.7374 | 0.7676 |
| 机器学习准确率提高方法（关键词相同） | 0.8062 | 0.9088 |
| 今天晚饭吃什么好（完全无关） | 0.4642 | 0.3505 |

结论：智谱embedding-3中文区分度更好，无关句相似度更低说明噪音更少。

### 为什么RAG用Embedding而不是关键词搜索
关键词搜索只能匹配相同的词，Embedding能理解语义——用户问「天气怎么样」，能检索到「今天阳光明媚」，即使一个字都不重合。

## 2. 向量数据库 FAISS

### 核心概念
FAISS是Facebook开发的向量检索库。它做的事情是：把大量向量存起来，给一个查询向量，快速找出最相似的top-k个。

**注意：FAISS只存向量，不存文字。** 文字要单独存在列表里，靠下标关联。

### 两种索引类型

**IndexFlatL2（暴力搜索）**
- 原理：把查询向量和库里每一个向量都算一遍L2距离，取最小的
- 优点：准确率100%
- 缺点：数据量大了很慢
- 适用：数据量小（几百到几千条），你的RAG项目用这个

**IndexIVFFlat（分区检索）**
- 原理：先把向量空间分成nlist个区域，检索时只在nprobe个最相关区域里找
- 优点：速度快
- 缺点：可能漏掉边界附近的相关结果，准确率略低
- 适用：数据量大（几十万条以上）

**你的实验数据：** 1万条向量，IVFFlat比FlatL2快2.7倍

### nprobe参数
控制检索时看几个分区。nprobe越大越准但越慢，nprobe=nlist时等同于FlatL2。

### L2距离 vs 余弦相似度
| | L2距离 | 余弦相似度 |
|--|--------|----------|
| 越相似时 | 值越**小** | 值越**大** |
| 计算方式 | 两点间直线距离 | 两向量夹角余弦值 |
| FAISS默认 | ✅ | 需要额外配置 |

## 3. 文档分块（Chunking）

### 为什么要分块
用户上传100页PDF，不能整个塞给模型：
- Context Window有限，放不下
- 检索精度下降，模型要在100页里自己找答案

分块后每次只取最相关的几段，又准又省token。

### RecursiveCharacterTextSplitter
LangChain提供的分块工具，「Recursive（递归）」的含义是分割有优先级：

```
先按段落（\n\n）切 → 切不够小再按句子（。）切 → 再按逗号切 → 最后按字符切
```

两个核心参数：

**chunk_size：** 每块的最大字符数。太小导致上下文不完整，太大引入噪音。

**chunk_overlap：** 相邻块之间的重叠字符数。目的是避免重要信息被切断在边界处。

```
无重叠：「张三是优秀工程师」 | 「他擅长Python」← "他"是谁不知道
有重叠：「张三是优秀工程师」 | 「优秀工程师他擅长Python」← 上下文完整
```

### 你的实验数据
同一份文档（约1000字）：

| 配置 | 块数 |
|------|------|
| chunk_size=100, overlap=0 | 8块 |
| chunk_size=200, overlap=50 | 4块 |
| chunk_size=500, overlap=100 | 2块 |

### 参数选择建议
| 文档类型 | chunk_size | overlap |
|---------|-----------|---------|
| 技术文档/学术论文 | 300-500 | 50-100 |
| 新闻/短文章 | 100-200 | 20-50 |
| 对话记录/FAQ | 按问答对切 | — |

## 4. 完整RAG Pipeline

### 流程图
```
【建库阶段】（只做一次）
文档 → 分块 → 每块转Embedding → 存入FAISS索引 → 保存到本地文件

【查询阶段】（每次提问触发）
用户问题 → 转Embedding → FAISS找最近的Top3块 → 拼成context
                                                        ↓
                                            context + 问题 → LLM
                                                        ↓
                                                    生成回答
```

### 索引持久化
每次重启程序都重新向量化很浪费API调用，解决方案是保存到本地：

```python
# 保存
faiss.write_index(index, "my_index.faiss")
pickle.dump(chunks, open("chunks.pkl", "wb"))

# 加载
index = faiss.read_index("my_index.faiss")
chunks = pickle.load(open("chunks.pkl", "rb"))
```

下次启动直接加载，不需要重新调用API。

## 5. 混合检索（Hybrid Search）

### 为什么需要混合检索
单一检索方式都有盲区：

| 检索方式 | 擅长 | 不擅长 |
|---------|------|-------|
| 向量检索 | 语义理解，同义词，模糊查询 | 精确术语、专有名词匹配 |
| BM25关键词检索 | 精确词语匹配，专业术语 | 同义词，语义理解 |

混合两者，取长补短。

### BM25 原理
基于词频的关键词检索算法。简单理解：查询词在某个文档里出现频率越高，这个文档得分越高。

中文使用时需要先用jieba分词：
```
「FAISS检索工具」→ jieba分词 → ["FAISS", "检索", "工具"]
```

### RRF融合（Reciprocal Rank Fusion）

**核心思想：用排名而不是分数来融合两路结果。**

原因：向量检索返回L2距离，BM25返回词频分数，单位完全不同没法直接相加。但排名是通用的。

**计算方式：**
```
每个文档的RRF得分 = 1/(k + 向量排名) + 1/(k + BM25排名)
k=60（经验值，防止第一名得分过高）

例子：某文档在向量检索排第1，BM25排第3：
得分 = 1/(60+1) + 1/(60+3) = 0.0164 + 0.0159 = 0.0323
```

得分越高越靠前，最终取Top3返回。

**优势：** 不需要归一化，对两路分数的量纲差异不敏感。

## 6. RAGAS 评估框架

### 为什么需要评估
没有量化指标就没有说服力。「感觉回答还不错」不能写进简历，「Faithfulness从0.62提升至0.85」才能。

### 四个核心指标

**Faithfulness（忠实度）**
- 测的是：回答里的每个说法，是否都有检索内容作为依据
- 计算方式：把回答拆成多个原子claim，用LLM判断每个claim是否在retrieved context里有支持
- 你的分数：1.0（满分），说明模型没有编造任何内容，完全基于检索结果回答

**Answer Relevancy（答案相关性）**
- 测的是：回答是否真正回答了问题（而不是答非所问）
- 计算方式：让LLM根据回答反推几个可能的问题，看这些问题和原始问题的相似度
- 你的分数：0.60，偏低原因是回答包含过多额外信息，不够简洁直接
- 优化方向：调整System Prompt，要求模型回答更简洁

**Context Recall（上下文召回率）**
- 测的是：检索到的内容，是否覆盖了回答问题所需的全部信息
- 你的分数：1.0，说明检索没有遗漏关键信息

**Context Precision（上下文精确度）**
- 测的是：检索到的内容，有多少是真正有用的（有没有检索到无关内容）
- 你的分数：1.0，说明检索结果质量很高，没有噪音
